The Seed Journal is a gardener's resource found at the-seed-journal.herokuapp.com. It was written from the ground up using React.js and the Trefle API by myself and two other students. My primary responsibility was to pull in the data from Trefle and create the template that the results page uses; I also set up the hosting and helped with the search bar and filter functionality. If the search function is not currently working, you can view the results page for the dog rose plant here.

The Seed Journal features:

  • Dark mode and light mode
  • A favorites list for plants
  • Search for a specific plant by name or find a random plant
  • Examine information on each plant on its result page
  • Filter results by sunshine and humidity required

Many plant informational websites show a very limited set of data, and we wanted to notify our users that the amount of data we deliver can vary from plant to plant. Therefore, we chose to represent missing information as "none listed", instead of simply removing the section in question from the results page.

Over the last few weeks, travel to and from the town of Darkmoor has trickled to a halt, save for a few refugees who whisper of necromancy and other horrors. You have been sent to find the truth. You arrive in Darkmoor as darkness falls. Your mission is simple: survive.

The Shadow Over Darkmoor is a text based roguelike game featuring over 100 locations to explore, maps, over 40 unique items to find, and over 30 enemy types. Darkmoor is written entirely in Clojure, a modern Lisp dialect, using functional programming.

I taught myself both Clojure and functional programming for this project because I wanted to experiment with using immutability and persistent data structures to avoid the state bugs and difficulty of updating I'd encountered while writing a similar game in C++. Of course writing a game, which is virtually nothing but side effects and state changes, in a paradigm that shuns both those things was difficult so early into my coding career!

A year later I completely revamped Darkmoor to include more functionality, smoother programming design choices, improved UI, containerized data structures, enemy stat scaling, and much more. To date Darkmoor remains the only project I've had the opportunity to redo.

Curry is a declarative programming language that was designed to integrate features from both functional programming and logic programming. This course was taught by one of Curry's designers, Sergio Antoy, who delved quite deeply into the theory behind the language features in Curry as well as its evaluation strategies and compilation. Much of the course focused on mastering the principals of nondeterminism, residuation, and narrowing, which are core features of Curry. Constriant programming was used to gradually reduce the scope of compilation during runtime due the ineffeciencies of pure nondeterministic programming.

This was one of the most rewarding courses I took during my Masters and Curry has become one of my favorite languages to work in due to its simplicity and power. I found constraint programming very engaging, and I'm excited to apply those principals to other areas. I also became comfortable with a more Haskell-like style of functional programming, as most of my prior experience in that area had been from Lisps.

Graveyard is a Halloween version of the game Banqi, originally designed by Thea Leake. The point of the game is to be the last player that can make a valid move, which can be done by either eliminating or trapping the other player's pieces. Thea's implementation of the game required both players to play at the same computer, so I decided to design and implement a networking protocol to add remote gaming to Graveyard.

Concurrency is inherent to Graveyard's design. To ensure safety, Thea chose to write Graveyard using Racket, a functional language that minimizes state changes and side effects. Though I was not familiar with concurrency before this project, I decided to utilize concurrent channels and message passing in both the client and the server code.

At a high level, the protocol relies on collecting the starting and ending coordinates of each valid move from one player and sending it to the other player. The server is responsible for game creation and management, ensuring clients are still connected, and validating and passing on client messages. Clients are responsible for ensuring only valid moves are sent to the server, and in enacting moves received from the server. Click the link below to view the RFC for more details.

In this technical report, myself and four other students present our research into race and gender biases in machine learning algorithms used in natural language processing, facial recognition systems, criminal justice software, and medical software. All of the algorithms we researched were rife with bias and inequality, with causes ranging from biased training datasets or design choices to a neglect to research the scientific backing of certain aspects of the algorithm before development. Consequences include the perpetuation of harmful stereotypes, an increased chance of targeted violence, unequitable crime sentancing, and the denial of medical care to women and racial, ethnic, and gender minorities.

Unfortunately, fixing the bias inherent in these systems is not an easy task for a variety of reasons. The lack of scientific backing during development, proprietary algorithms, general lack of awareness, or the support of these systems within certain industries all play a part.

I wrote and researched section 3, Gender and Racial Bias in Facial Recognition Systems.

Oct 2019 - Dec 2019 | Solo

This paper began as an experiment for PSU's Code Reading and Review course to see if I found typed Lisps easier to read than untyped Lisps. Many typed Lisps use the Hindley-Milner typing system, which I was biased against due to the dislike of Haskell I had at that time. In an effort to debias myself I spoke with one of the designers of Haskell, Mark Jones, about the rationale behind the inclusion of the Hindley-Milner typing system as one of the core features of Haskell. When Mark informed me that the Hindley-Milner typing system was partly inspired by Lisp, I decided to make that relationship the focus of my research instead.

In summary: Lisp pioneered dynamic typing, as static typing could not support the flexibility inherent to Lisp. Dynamic typing, however, comes with certain drawbacksâ€”namely that typing errors are often not found before runtime and can instead crop up as "rather inscrutable bugs", as Henry Milner put it. These frustrations led to efforts to develop a typing system that supported flexible programming, but would be strict enough to avoid runtime type errors. That effort eventually culminated in the Hindley-Milner typing system, which was first described by Roger Hindley and later rediscovered by Milner.

Instead of introducing this typing system to Lisp, Milner built a new language around it. This language was ML, which has spawned several popular programming languages used today such as Haskell and Ocaml. It was only after several decades that the popularity of these languages lead to the creation of a popular typed Lisp: typed Racket.

May 2019 - Jun 2019 | Solo

This database contains some of the spells used in the 5th edition of Dungeons and Dragons. The schema design, implementation, and data entry were all done by myself. D&D spells are elaborately interconnected to other areas of the game, which made defining the relations of the data and designing the schema a challenge. This was the first project I completed during my Masters program, and I learned a lot about importance of having the proper scope and realistic goals for a project.

Viewed in isolation, the database schema is overly complicated. I wanted the design to support the addition of non-spell data, but this split my focus and I failed to notice certain patterns in the design of the spells that would have simplified the schema I used. As a result, certain queries became longer and more cumbersome to write than they otherwise needed to be. In practice, however, this effect could be largely mitigated with the addition of a user interface (such as a search form or an API) to streamline query creation.

This FTP client was written by a team using Agile development practices as part of the Agile Software Development course at PSU. Our FTP client is written in Python, and uses Pytest for unit testing and Pylint to ensure PEP8 guidelines are followed. The FTP client aims to replicate the native Unix ftp client as much as possible, and only works in Unix/OSX systems.

I set up a FTP server in AWS which our client connected to. I also wrote the following code:

  • Connect to the server
  • Log in and out
  • Contributed to time-out functionality
  • Commands: move directory, remove directory, change file permissions
  • Validate user input (our client accepts nearly all of the file names and formats Unix supports)
  • Command line interface, including code to handle mistypes and server errors
